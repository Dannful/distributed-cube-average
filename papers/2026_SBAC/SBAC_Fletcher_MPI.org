** Comm. Cost vs. Partitioning Strategy
#+begin_src R :results output :session *R* :exports both :noweb yes :colnames yes
# Load libraries
library(ggplot2)
library(tidyr)
library(dplyr)

# Define the data
P <- seq(1, 5010, by = 10) # Number of processors
L <- 1 # Normalized side length

# Internal Surface Area Formulas
area_1d <- 2 * (P - 1) * L^2
area_2d <- 4 * (sqrt(P) - 1) * L^2
area_3d <- 6 * (P^(1/3) - 1) * L^2

# Create a data frame and convert to long format for ggplot
df <- data.frame(Processors = P,
                 `Slices` = area_1d,
                 `Columns` = area_2d,
                 `Blocks` = area_3d)
df_long <- pivot_longer(df, cols = -Processors,
                        names_to = "Partitioning_Type",
                        values_to = "Boundary_Area") |>
  mutate(Partitioning_Type =
           factor(Partitioning_Type,
                  levels = c("Slices", "Columns", "Blocks"),
                  labels = c("Slices (1D)", "Columns (2D)", "Blocks (3D)")
                  ))

# Generate the plot
ggplot(df_long, aes(x = Processors, y = Boundary_Area, color = Partitioning_Type)) +
  geom_line(size = 0.8) +
  labs(#title = "Communication Cost vs. Partitioning Strategy",
       #subtitle = "Comparing 1D, 2D, and 3D Decomposition for a Cube",
       x = "Number of Processing Elements (P)",
       y = "Internal Boundary Area\n(Comm. Volume in\ncommunicated items)",
       color = "Partitioning Strategy") +
  theme_bw(base_size=8) +
  coord_cartesian(ylim = c(0, 200)) +
  scale_color_manual(
    values = c("red", "blue", "darkgreen"),
    labels = c("Slices (1D)", "Columns (2D)", "Blocks (3D)")
  ) +
  scale_x_continuous(expand = c(0.01, 0)) +
  scale_y_continuous(expand = c(0, 0)) +
  theme(
    plot.margin = unit(c(0, 0.2, 0, 0), "cm"),
    legend.spacing = unit(0.1, "mm"),
    panel.grid = element_blank(),
    legend.position = "top",
    legend.direction = "horizontal",    
    legend.justification = "left",
    legend.key.size   = unit(0.2, "cm"),
    legend.box.spacing = unit(0, "pt"),
    legend.box.margin = margin(0, 0, 0, 0)
  ) -> p
ggsave("img/comm_cost.png", width = 4, height = 1.3, units = "in", dpi = 300)

** Introduction
*** Context                                                        :ignore:

Halo exchange is a foundational technique for distributing iterative
stencil computations across HPC nodes. Each MPI process maintains a
/ghost-cell/ layer, which is a local copy of its neighbors' boundary
data, and asynchronously overlaps these transfers with interior
computation. Whether this overlap achieves genuine communication
masking depends on the interplay of problem size, partitioning
topology, and interconnection bandwidth: with small domains or slow
networks, communication dominates; with large domains and fast
networks, computation can fully hide the transfers
\cite{stencil_supercomputers}. When correctly tailored to the variety
of configurations, the benefits of halo exchange are undeniable.

*** Present the existing Fletcher implementation and RW            :ignore:

Fletcher solves pseudo-acoustic wave equations for seismic imaging in
anisotropic media \cite{fletcher, fletcher_modeling}, applying an
eighth-order finite-difference stencil over a discretized 3D grid at
each time step.  The application is computationally intensive and
memory-bound per iteration, and benefits strongly from GPU
acceleration \cite{multi-gpu, okamoto2010}.

Early contributions targeted GPU memory efficiency on single-node
systems. Pavan et al. applied shared memory, read-only cache, and
register reuse strategies to Fletcher-based stencil kernels, achieving
performance improvements of up to 264.6% on NVIDIA P100
\cite{pavan2018wperf}.  I/O was identified as a complementary
bottleneck, and checkpointing strategies reduced RTM execution time by
up to 17.33% \cite{pavan2018wscad}.  A significant algorithmic advance
was the reformulation of the Fletcher cross-derivative terms, reducing
floating-point operations and yielding up to 73.6% performance and
42.6% energy efficiency improvements \cite{serpa2020erad}; these
optimizations were subsequently integrated into a systematic
comparison across multicore and GPU architectures using OpenMP,
OpenACC, and CUDA, establishing that CUDA on NVIDIA Volta delivers
peak performance while OpenACC provides a portable alternative with
less than 10--20% degradation \cite{serpa2019wscad}
\cite{serpa2021ccpe}.  At the multi-GPU level, Rigon et al. overlapped
inter-device border-region communication with computation, reducing
the energy-delay product by up to 40% \cite{rigon2024erad}, while
Schussler et al. proposed a genetic-algorithm-driven autotuner (BTO)
for CUDA thread-block configurations that achieves up to 83.8% EDP
improvement over default execution on both AMD and NVIDIA GPUs
\cite{schussler2024pdp}.  Portability to novel architectures has also
been investigated. Künas et al. evaluated SYCL with ESIMD and subgroup
optimizations for the Fletcher/RTM kernel on Intel Max GPU, obtaining
speedups of up to 4.6\times \cite{kunas2025carla}, and Lorenzon et
al. assessed twelve programming-model implementations across eleven
GPUs from NVIDIA and AMD, reporting that HIP and RAJA achieve the
highest carbon-efficiency portability \cite{lorenzon2025sbacpad}.  At
the systems level, Machado et al. deployed the Fletcher application on
cloud SaaS instances \cite{machado2023sbacpadw}, and introduced
MPI-based strategies to overlap computation with disk I/O, achieving
speedups of up to 1.85\times \cite{machado2026ccpe}.

Several related efforts address halo exchange and stencil
communication from complementary angles. Pearson et
al. \cite{stencil_supercomputers} show that 3D block decomposition
minimizes the total communication boundary for Cartesian stencil
applications on heterogeneous supercomputers, and develop a node-aware
MPI+CUDA assignment strategy that maps subdomains to GPUs based on
inter-GPU communication bandwidth. Okamoto et al. \cite{okamoto2010}
accelerate seismic wave propagation on multi-GPU clusters with domain
decomposition and ghost-point exchanges, observing that communication
overhead scales as $N^{2/3}$ with the number of GPUs and that small
domains suffer severely from network latency, motivating the need for
large problem sizes. Van den Berg et al. \cite{gmd_halo} study MPI
halo exchange patterns for atmospheric models using SST/macro and
SimGrid, but with a focus on the scalability of collective patterns
rather than computation-communication overlap in stencil
kernels. Mercier and Jeannot \cite{simdag_affinity} evaluate process
placement for MPI applications with halo-exchange communication,
centering on topology-aware optimization of process affinity. Wittmann
et al. \cite{blocking_nonblocking_halo} benchmark blocking versus
non-blocking halo exchange for Lattice Boltzmann methods on real
supercomputers, confirming that overlap is achievable but inherently
hardware-specific. In the context of seismic applications, as
previously explicited, Rigon et al. \cite{multi-gpu} distribute the
Fletcher computation across multiple GPUs within a single node using
device-to-device ghost-cell transfers, achieving a speedup of 2.77 for
large domains but observing severe degradation on small grids due to
synchronization overhead.

Despite this extensive body of work, all existing multi-GPU Fletcher
implementations \cite{multi-gpu, okamoto2010} are limited to a single
computational node, operating within a single compute node or within a
multi-GPU server relying on shared memory or peer-to-peer GPU
links. This single-node limitation imposes immediate scalability
constraints in both computational throughput and GPU memory. The
natural path to resolving this bottleneck is to distribute computation
across multiple nodes via proper domain partitioning and inter-node
asynchronous communication.  As far as we known, the distribution of
the Fletcher method implementation across multiple cluster nodes via
MPI, with ghost-cell halo exchange to manage subdomain boundary data,
has not been explored. There is a clear gap in the literature leaving
open the question of how partitioning topology and network bandwidth
jointly determine communication-overlap effectiveness at large-scale
cluster deployments for this application. Although the implementation
itself of the halo exchange remains relatively easily attainable, the
challenges relies really taking advantage from this implementation in
realistic scenarios. For example, when paired with a slow
interconnect, it creates precisely the bandwidth asymmetry that makes
ghost cell masking most challenging to achieve. GPU resources finish
local stencil computation far faster than the network can deliver
boundary data. It becomes necessary therefore not only design the halo
exchange but carry out a profound performance analysis to tackle these
challenges.

The contributions of this paper are: *(i)* a distributed MPI
implementation of the Fletcher application with 3D block decomposition
and asynchronous ghost cell communication, numerically validated
against the single-node reference implementation across several
configurations; *(ii)* GPU acceleration via CUDA with wavefield data
permanently resident in device memory, preserving the same
computation-communication overlap strategy as the CPU backend; and
*(iii)* an evaluation on real HPC clusters showing how network bandwidth
and partitioning topology jointly determine ghost cell masking
effectiveness, with compute loads ranging from approximately 20% on a
1Gbps GPU cluster to above 87% on a 10Gbps CPU cluster.  These
contributions collectively establish that ghost cell effectiveness is
inherently application- and platform-dependent, and scales with
problem size, partitioning shape, and available bandwidth.
